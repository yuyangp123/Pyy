# 睡眠和恢复(多源融合)

> 从 Apple Watch / HAE 原始数据,经多源融合算法,得到比手表更准的真实睡眠判断,并可视化。
> 案例夜:2026-06-10 夜 → 06-11 晨。取数 / 格式 / 解析见 `file-operations.md`。

## 0. 背景与目标

**痛点 — 单一数据源都会骗人:**
- HAE JSON 聚合值在刚醒时导出会 **低估**(曾导出 5.35h——导出脚本在醒后几分钟跑、Apple 还没定稿分期)。
- Apple Health 截图(= Apple Watch)会 **高估**(显示"睡了 8h01m")。
- 手表把"躺着看球"误判成睡眠。

**目标:** 读到分时段睡眠 + 多源交叉验证 → 可信的真实睡眠时长与结构。

## 1. 数据层

### 1.1 让 HAE 导出分段
- HAE 默认把睡眠聚合成整夜总量,不导分段。
- 解法:新建 **sleep-only 自动化** + 关掉 **"Aggregate Data"** → 每个阶段段落一行
  (`start / end / value(阶段) / source / qty`)。
- ⚠️ 只对睡眠关聚合,别全局关(否则 HR/步数原始样本会把文件撑爆)。

### 1.2 两个数据源(关键发现)
导出里同时有两套睡眠记录:
- **Apple Watch**(原生,HR + 加速度计分期)—— **漏报清醒**。
- **Sleep Cycle**(App,声音 + 体动)—— **检出更多清醒**。
两者时间重叠、会打架,需融合。

### 1.3 两种文件,各取所长

| 文件 | 内容 | 用途 | 局限 |
|---|---|---|---|
| `HealthAutoExport-*.json`(分段) | 每段 start/end/stage/source | 精确 hypnogram | 34KB,base64 搬运不可靠 |
| `HealthMetrics-*.csv`(聚合) | 整夜各阶段总量 | **精确阶段总量** | 无分段时序 |

> 实战折中(已升级):**6/11 起 HealthMetrics CSV 已含逐分钟 HR + 呼吸率**,
> 事件定位从"逐时"升到"逐分钟"。分段 JSON(阶段时序)仍过大、base64 搬运失真——
> 逐 stage-epoch 时序若要,仍需 HAE 输出**分段 CSV**(体积小)。

### 1.4 案例夜精确数据(HAE CSV,与在床对账通过)
- 在床 **9h53m**(22:54→08:47)
- Core **268m** / Deep **88.5m** / REM **45m** / Awake **191m(3h11m)**
- 设备合并 TST = **6.69h**;校验:Stages + Awake = 在床 ✓

### 1.5 自主神经(逐分钟 HR + 呼吸率,来自 HealthMetrics CSV)
- HRV(**SDNN**,非 rMSSD)逐时;剔除 >2× 伪迹(01:00=135、08:00=160,无对应 HR 升高 = 伪迹)。
- HR **逐分钟**(125 点):深睡谷底 **39.4 @ 02:42**、看球段 **均 54**、清晨升至 **68**。
- **呼吸率**(40 点):深睡 **14** → 看球 **17** → 重新入睡 **15**;与 HR **同步抬升 = 独立佐证清醒**
  (研究:呼吸率专补强清醒/REM 判定)。
- ⭐ HRV 取法 = `download_file_content` HRV Export JSON → base64 解码 → SDNN 逐时,
  和 HR 同一时间轴 plot(见 `file-operations.md`)。

## 2. 研究地基(Evidence Base)

### 2.1 消费级设备分期偏差(PSG 验证)
- **Apple Watch**:睡眠灵敏度高(~98%),但**清醒特异度低(~50%**,Apple 自报 76%);
  系统性**高估 TST ~39min、低估 WASO ~45min**;低估深睡(只认 ~50%、−43min)、
  高估浅睡(+45min);Cohen's κ≈0.53。
- 加速度计会把**"安静仰卧清醒"(躺着看电视/看书)误判成睡眠** ← 看球正中此坑。

### 2.2 自主神经 ↔ 睡眠分期
- 副交感随睡眠加深增强(N1→N2→N3);**HR 在深睡最低、随 NREM 逐步下降、觉醒时达峰**;
  清醒 HR 介于 NREM 与 REM 之间。
- **REM 的 HR ≈ 清醒** → 光靠 HR 分不开 REM/清醒,需 HRV + 体动 + 时段。
- 纯 HRV 分期可达 κ≈0.61(与消费设备相当)。

## 3. 算法:自主神经仲裁的多源融合

**核心思想:** 两个设备分期 = 有偏差的投票;**HR/HRV = 生理裁判**,对"清醒 vs 睡眠"有最终判罚权
(设备最弱项);设备只决定睡着时的"哪个阶段"。

**5 步管线:**
1. **统一 epoch 网格**(30s);设备无数据处(手表 gap)标 NaN。
2. **造唤醒指数** Arousal(t) ∈ [0,1],**按当晚自身归一化**(HRV/HR 高度个人化):
   `Arousal = 0.7·z(心率) + 0.3·z(HRV下塌)`,心率权重更高(更稳的觉醒标志)。
3. **两级判罚:**
   - 清醒 vs 睡眠(自主神经主导):Arousal 持续高(≥5min)→ 判**清醒**,即使两设备都说睡 → 救出看球段。
   - 睡着分阶段(设备主导 + 自主神经微调):低 HR + 高稳 HRV + 规律呼吸 → 提**深睡**;
     HR 高 + HRV 未塌 + 清晨 → **REM**。
4. **生理平滑**:HMM/Viterbi + 最小段长(去单 epoch 噪声;清醒→深睡不可直接跳,必经浅睡)。
5. **输出**:融合 hypnogram + TST/效率/WASO + 每段置信度(三源一致度)。

> ⚠️ 陷阱:REM 的 HR ≈ 清醒,必须用 HRV 是否塌 + 体动 + 时段来区分,否则会把 REM 误判成清醒。

本质 = 把 HRV-LSTM 那类(κ≈0.61)机器学习做法,改写成**可解释、能跑在稀疏数据上的规则版**。

## 4. 判断(案例夜应用结果)

### 三源对照

| 来源 | TST | 清醒 | 说明 |
|---|---|---|---|
| Apple Watch 单独 | 8.0h | ~1.9h | **高估**,漏报看球清醒 |
| 双源合并 | 6.7h | 3.18h | Sleep Cycle 检出更多清醒 |
| **融合判断** | **~6.9h** | **~3h** | 自主神经佐证 |

### TST 估计收敛轨迹(每次用更好数据 → 显式纠错)
`6–6.5h(纯目测)` → `7.3h(自主神经+文献两条路)` → **`6.9h`(精确 Awake 3h11m 锚定)**

### 结构(精确)
- 深睡 **88.5m(22%)✓ 达标** — 前半夜身体修复保住。
- **REM 45m(11%)✗ 偏低**(正常的一半)— 看球(03–05)吃掉凌晨 REM 富集段。
- 效率 **~70%**(干净夜应 85%+)。

### ⭐ 结论
**损失偏认知/情绪巩固(REM),非体能修复(深睡)。** → 今晚补后半夜(REM 在凌晨),次日守 8h。

### 时间轴交叉印证(单看一张图看不出)
- **02:00** 深睡 ↔ HR 最低 39 + HRV 84。
- **03–05** 手表记成睡眠,但 HR↑55 + HRV↓50 → 揭穿为清醒(已校正)。
- **07:00** HR 高但 HRV 未塌 → REM 非清醒。

## 6. SOP & 经验(给未来的自己)

- **睡眠 ground truth = 截图**,但 Apple Health Sleep 视图 = Apple Watch,**会高估睡眠**(漏报清醒);
  HAE 聚合在刚醒导出会**低估**。两个都别单独信。
- **精确总量**:用 HAE **HealthMetrics CSV**(聚合但精确,Stages+Awake 对账在床)。
- **精确分段**:HAE **sleep-only + 关 Aggregate** → JSON(但大,搬运不可靠;要逐 epoch 就改**分段 CSV** 输出)。
- **两源必须融合**:AW 漏报清醒、Sleep Cycle 过报清醒,真实在两者间;别只信手表的 8h。
- **自主神经仲裁**:某段"睡眠"若 **HR↑/HRV↓ = 大概率清醒**;用 HR 反查揪出误判睡眠。
- **HRV**:用夜间 **SDNN**,剔 >2× 伪迹(配合 HR 判断),对比个人基线 **65–85**。
- **数据反模式**:别目测先算;n=1 非因果;跨午夜务必确认日期。
